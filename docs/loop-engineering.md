# Loop Engineering × 短篇小说创作

## 什么是 Loop Engineering？

用大白话说：**把一个复杂任务拆成多个有边界的循环，每个循环只做一件事，循环之间通过明确的产物和批准点衔接。** 就像工厂流水线——上游产出的合格件才进入下游，有问题打回去修，修完重新过质检。

Loop Engineering 来自 AI Agent 系统的设计范式，但它要解决的问题和写短篇天然重叠：

| Loop Engineering 概念 | 写作中的对应 |
|---|---|
| **Loop（循环）** | 一个独立的创作阶段——选题、审稿、改稿、终校 |
| **产物（artifact）** | 该阶段产出的文件——spec、draft、review report |
| **批准点（approval gate）** | 作者确认后才进入下一阶段 |
| **质量门（quality gate）** | 不达标自动回炉，达标自动推进 |
| **预算（budget）** | 最大修改轮数，防止无限循环 |

```
                   │ spec_approved

                   │ architecture_approved
                   ▼

└──────────────────┬──────────────────────────────┘
                   │ 写入 draft
                   ▼
┌─────────────────────────────────────────────────┐
│              内容审稿 Loop                        │
│  4 个盲审 Agent → 聚合评分 → 自动爬坡             │
│  最多 3 轮修复 + 2 轮优化                        │
└──────────────────┬──────────────────────────────┘
                   │ quality_ready
                   ▼
┌─────────────────────────────────────────────────┐
│              通顺性终校 Loop                      │
│  3 个文字医生 → 1 个裁决 → 1 个改稿 → 复验        │

└──────────────────┬──────────────────────────────┘
                   │ publication_ready
                   ▼

```

**五个 Loop，层层递进。** 每个 Loop 有清晰的"进去什么、出来什么、什么算过、什么算不过"。

## 核心设计决策

### 1. 盲审 = Loop 内的并行 sub-agent

内容审稿 Loop 不是一个人看完打分——是**四个独立的 sub-agent 同时读同一篇稿子**，每人只管一个角度。

这直接对应 Loop Engineering 的核心模式：**一个协调器（orchestrator）派发多个 leaf agent，汇总结果后决定下一步。** 四个 sub-agent 互不知晓——协调器分别给每个 agent 不同的上下文（因果 agent 甚至看不到预设结局），拿到四份独立报告后聚合评分和路由。

同样，通顺性终校 Loop 里也有 sub-agent 模式：三个文字医生（syntax / coherence / readaloud）并行审稿 → 一个裁决 agent 合并问题 → 一个改稿 agent 统一执行。

这样设计的目的：

- **防止从众**。因果审稿不知道人物审稿给了什么分，不会因为"别人给得高我也给高"
- **防止信息泄露**。因果审稿甚至不知道故事的预设结局——它是真正的盲审
- **聚合更可靠**。四个独立判断的聚合比一个人的全面判断更接近客观评价

### 2. 爬坡 = 有安全保证的自动迭代

传统的"写了改、改了写"有两个风险：(a) 越改越差，(b) 不知何时停。

爬坡机制解决了这两个问题：

- **分数不升则保留最佳**。每一轮改完重新审稿，分数不升反降就保留上一版最佳稿的路径，所有后续环节使用最佳稿
- **预算耗尽自动停**。修复最多 3 轮，优化最多 2 轮，通顺性最多 2 轮——耗尽即以当前最佳稿进入下一阶段
- **路由基于当前维度弱点**。系统根据本轮审稿中最弱的维度决定下一轮改什么方向，不记录历史成功率

### 3. 分层评分 = 承认不确定性

不是所有评分维度都一样可靠。系统把它们分成三层：

```
硬门  因果 / 递进 / 兑现        → 客观可证，< 4 不过
软门  文字 / 余味 / 开头        → 可论证，< 4 不过但降级
主观  差异化 / 能动性 / 情绪    → 审美分歧，仅提醒
```

这相当于说：**"故事逻辑断了就是断了，没商量。但'题材够不够独特'——不同读者有不同答案，不给满分可以，别因为这个卡住一篇好故事。"**

### 4. 产物驱动 = 每个阶段有可见的交付物

没有"我觉得差不多了"。每个 Loop 产出一个**具体文件**：

```
story_spec.yaml       → 选题结论
loop_map.json         → 叙事架构
drafts/draft-vN.md    → 第 N 轮稿子
reviews/review-vN/    → 第 N 轮审稿报告（4 份 + 聚合）
copyedit/round-N/     → 第 N 轮终校报告（3 份 + 裁决 + 聚合）
final.md              → 最终可发表稿
```

作者可以在任何时间点翻阅任何一轮的审稿报告，看 AI 当时怎么评价、怎么改的。

## 版本历史

### v1 — 循环叙事专用（816 行）

最初的版本。只支持"时间循环/重复事件"类短篇（Groundhog Day 式的）。基础的结构化评分，人工决定何时修改、何时停下。

### v2 — 多题材 · 多 Agent 审稿（2420 行）

**这一版确立了整个架构。**

- 放开题材限制，支持任何类型的短篇
- **四角色盲审**上线：因果（盲审）、转化、人物、文字——四个互不知晓的 AI 同时审稿
- **连续性台账**：追踪故事中人物身份、时间线、关键道具的一致
- **替代解释反证**：每个叙事闭环必须排除同等强度的竞争解释，防止"没写清楚假装留白"
- **9 维 45 分制**建立
- **通顺性终校**：三审（语法/逻辑/朗读）一裁一改

### v3 — 双阶段自动爬坡（3730 行）

不再需要作者手动决定"改不改、从哪改"。内容和通顺性都变成**有预算控制的自动优化**。开放结局不再被误判为 bug——`remaining_uncertainty` 声明让系统区分"有意留白"和"写糊了"。

### v4 — 智能评分 · 规则系统（4439 行，当前版本）

**让评分更聪明，让规则可声明。**

- **HARD / SOFT / SUBJECTIVE** 三层评分：审美分歧不卡好稿子
- **`world_rules`**：作者声明故事中的超自然规则，审稿时检查正文是否违规
- **`mirror_character`**：追踪与主角对位的关键人物
- **`protagonist_behavior_is_credible`**：有专业身份的主角行为是否与身份矛盾
- **最小闭合检查**：开放式结尾至少有一样东西是确定的
- **`decide-quality` / `reapprove-premise`**：极端情况下作者可手动放行
- **评分波动检测**：同类 Agent 批间评分差异过大时标为提醒
- **150+ 自测项**确保每次修改不退化

## 与纯 AI 写作的区别

| | 让 AI 写一篇 | loop-short-story |
|---|---|---|
| 反馈 | 你读完觉得不好，说不清哪不对 | 四个 Agent 分别告诉你每个维度的分数和证据 |
| 修改 | 手动改，改完不确定进步没 | 自动爬坡，分数不涨自动滚回 |
| 一致性 | 每次让 AI 评价标准不同 | 确定性脚本 + 标准化审稿模板 |
| 可追溯 | 改来改去丢了原始稿 | 每轮稿子和审稿报告完整存档 |
